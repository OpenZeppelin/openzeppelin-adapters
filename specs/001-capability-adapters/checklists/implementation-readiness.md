## Scenario Coverage — Edge Cases

- [x] CHK038 — Are requirements defined for what happens when a consumer imports both a profile runtime AND a standalone capability from the factory map for the same `NetworkConfig`? Are they isolated or should they share state? [Edge Case, Gap] — **Addressed**: Added to spec edge cases — "They are fully isolated — the standalone capability does not share state with the profile runtime."
- [x] CHK039 — Is the behavior specified when `createRuntime` is called with an invalid `ProfileName` (not one of the 5 defined names)? [Edge Case, Gap] — **Addressed**: Added to spec edge cases — "MUST throw a `TypeError` with a message listing the valid profile names."
- [x] CHK040 — Are requirements for concurrent `dispose()` calls defined — is it idempotent, or does double-dispose throw? [Edge Case, Spec §FR-010] — **Addressed**: Spec edge cases and data model both state "`dispose()` is idempotent — calling it twice is a no-op (does not throw)."
- [x] CHK041 — Are requirements defined for adapter packages that implement a capability but only partially (e.g., `ExplorerCapability` with `getExplorerUrl` but not `getExplorerTxUrl` which is optional)? [Edge Case, Contracts §ExplorerCapability] — **Addressed**: Capability interfaces use TypeScript's optional method syntax (`method?()`) for optional methods. An adapter satisfies the interface by implementing all required methods; optional methods may be omitted. This is standard TypeScript interface compliance.
- [x] CHK042 — Are requirements for error messages specified when a consumer accesses a capability that exists on the runtime type but was not included in the requested profile (TypeScript type vs runtime availability)? [Edge Case, Gap] — **Addressed**: Added to spec edge cases — "The property is `undefined` at runtime. TypeScript's type system reflects this via optional properties on profile-specific runtime types."
